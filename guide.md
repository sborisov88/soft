# Руководство по изучению: Работа с частичными стендами ДБО

Данное руководство составлено на основе технического обсуждения процесса развертывания, тестирования и эксплуатации «частичных стендов» (Particle Stands) в среде ДБО. Документ предназначен для глубокого изучения архитектуры, процедур деплоя и особенностей взаимодействия сервисов в изолированных тестовых пространствах.

---

## 1. Основные концепции и архитектура

### Принцип работы частичного стенда
Частичный стенд представляет собой изолированное пространство имен (namespace), в котором разворачиваются только необходимые для тестирования сервисы (обычно от 1 до 5). 
*   **Логика запросов:** Если вызываемый сервис развернут на частичном стенде, запрос направляется именно туда. Если сервис отсутствует в локальном пространстве имен, система автоматически перенаправляет запрос на основной стенд (QF) через механизмы DNS и `External Name`.
*   **Интеграция:** Частичные стенды используют общие ресурсы основного стенда (базы данных, секреты Kafka, RabbitMQ), что позволяет экономить ресурсы и упрощать конфигурацию.

### Процесс развертывания (Deployment)
1.  **Подготовка:** Выбор задачи (тикеров) и определение сервиса (например, `Cash Credit`).
2.  **Конфигурация:** Внесение изменений в `Properties` через интерфейс `Customize`, создание коммита или использование готовой ветки.
3.  **TeamCity:** Выбор билда (обычно из ветки `master`), указание параметров деплоя и запуск `Run build`.
4.  **Автоматизация:** Процесс управляется Python-скриптами, которые через REST API запускают последовательные и параллельные задачи (создание namespace, установка политик, копирование SVC, деплой конфигураций и самих образов). Время развертывания составляет около 7–10 минут.

### Жизненный цикл стенда
*   **Удаление по неактивности:** Стенд автоматически удаляется, если он не используется более 48 часов. 
*   **Механизм очистки:** Ежедневно в 18:00 запускается скрипт, проверяющий дату последнего билда/деплоя. Если разница с текущим временем превышает 48 часов, стенд уничтожается.
*   **Ручное управление:** Предусмотрена специальная джоба в TeamCity для принудительного удаления стенда.

## 2. Практические вопросы для самопроверки (Short-Answer Quiz)

1.  **Как формируется имя частичного стенда для доступа из приложения?**
    *Ответ:* Имя формируется по шаблону: `DBO_Particle_[Имя_пользователя_TeamCity]`.

2.  **Что произойдет, если запросить сервис, который не был деплоился на частичный стенд?**
    *Ответ:* Запрос будет перенаправлен на основной стенд (QF).

3.  **В чем заключается главная проблема при тестировании асинхронных сервисов (Kafka/Rabbit) на частичных стендах?**
    *Ответ:* Возможна «гонка» данных и их потеря, так как частичные стенды используют те же consumer-группы и адреса, что и основной стенд.

4.  **Какое действие необходимо предпринять, если в мобильном приложении отображается статус "Particle Not Auth" или "No Auth"?**
    *Ответ:* Необходимо убедиться, что стенд уже поднят, установлены доверенные сертификаты, а также выгрузить приложение из памяти и авторизоваться повторно.

5.  **Как можно предотвратить автоматическое удаление стенда через 48 часов, если тестирование еще не завершено?**
    *Ответ:* Необходимо выполнить передеплой любого сервиса или запустить билд на стенде, чтобы обновить метку времени последнего деплоя.

6.  **Где хранятся общие настройки (сертификаты для Kafka/Rabbit) при создании нового namespace для стенда?**
    *Ответ:* Они хранятся в общих ConfigMap, которые монтируются к подам при запуске.

## 4. Глоссарий важных терминов

| Термин | Описание |
| :--- | :--- |
| **QF (Quality Fusion)** | Основной тестовый стенд ДБО, являющийся базой для частичных стендов. |
| **Частичный стенд (Particle Stand)** | Динамически создаваемое изолированное окружение (namespace) для тестирования отдельных фич. |
| **Namespace (Пространство имен)** | Логическая изоляция ресурсов в кластере, используемая для разделения стендов разных пользователей. |
| **Consumer Group** | Идентификатор группы потребителей в Kafka; использование одинаковых ID на разных стендах приводит к конфликтам вычитки данных. |
| **External Name** | Тип сервиса в Kubernetes, позволяющий перенаправлять трафик на внешние DNS-имена (в данном случае — на QF). |
| **TeamCity Agent** | Вычислительный ресурс, на котором выполняется сборка и деплой; может автоматически отключаться (автоскейлинг), вызывая сбои. |
| **SVC (Service)** | Объект инфраструктуры, обеспечивающий сетевой доступ к поду (приложению). |
| **ConfigMap** | Объект, используемый для хранения конфигурационных данных, которые монтируются в контейнеры. |
